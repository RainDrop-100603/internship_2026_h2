# 작업 단위 규칙 — 폴더 + 브랜치

> 하나의 구분되는 작업(work phase)을 **전용 폴더 + 전용 브랜치**로 진행하는 규칙.
> 요약은 [CLAUDE.md](../CLAUDE.md)에 있고, 이 문서가 상세 기준이다.

## 규칙

### 1. 폴더 — 작업마다 번호 붙은 전용 폴더

- 작업마다 번호를 붙인 폴더를 새로 만든다 (예: `1_사전공부/`, `2_XXX/`)
- 번호는 작업 시작 순서대로 증가
- 새 폴더를 만들면 CLAUDE.md의 폴더 구조에 반영

### 2. 브랜치 — 작업 시작 시 main에서 분기

- 브랜치명 형식: `work/<번호>-<영문-슬러그>`
- 폴더와 브랜치는 1:1 대응 (예: 폴더 `2_ABAP기초학습` ↔ 브랜치 `work/2-abap-basics`)
- 브랜치명에 한글을 쓰지 않는 이유: git 동작 자체는 문제없으나 터미널·GitHub URL에서 깨져 보이는 경우가 있음

### 3. 커밋 — 작업 중 모든 커밋은 해당 브랜치에서

- 공용 문서(`docs/`, `CLAUDE.md`, 루트 `README.md` 등) 수정도 **그 작업에서 발생한 변경이면 같은 브랜치에서** 진행
- 이렇게 하면 공용 파일의 변경이 어느 작업에서 왔는지 merge 커밋 단위로 추적 가능

### 4. merge — 작업 종료 시에만, 반드시 `--no-ff`

```
git checkout main
git merge --no-ff work/2-abap-basics
```

- **fast-forward 금지** — ff로 merge하면 브랜치 커밋이 main 히스토리에 그대로 섞여 "이 작업에서 무엇이 바뀌었나"의 경계가 사라짐
- `--no-ff`로 만든 merge 커밋이 작업의 경계 역할을 함:
  - `git log main --first-parent` → main이 **작업 단위 목록**으로 읽힘
  - merge 커밋 안으로 들어가면 해당 작업의 상세 커밋 확인 가능

### 5. 보존 — merge 후에도 브랜치는 삭제하지 않는다

- 작업별 기록 유지가 목적 (브랜치 = 작업의 이름표)
- ⚠️ GitHub에서 PR로 merge하는 경우 브랜치 자동 삭제 옵션이 켜져 있을 수 있음 → 삭제하지 않도록 주의

### 6. 예외 — 소규모 정비는 main 직접 커밋 허용

- 오타 수정, CLAUDE.md·규칙 문서 정비 등 **어느 작업에도 속하지 않는** 소규모 변경은 main에서 직접 커밋 가능
- 판단 기준: "이 변경이 특정 작업 폴더의 산출물인가?" — 아니면 main 직접 커밋 OK

## Claude 작업 지침

- 새 작업 폴더를 만들어 진행할 때는 **브랜치 생성부터 시작**한다 (`git checkout -b work/<번호>-<슬러그>` from main)
- 작업 브랜치가 존재하는 동안 main에는 해당 작업의 커밋을 만들지 않는다
- 작업 종료 merge는 사용자 확인 후 `--no-ff`로 수행한다
